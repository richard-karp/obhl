  Your question: will the import work with any esportsdesk site?

  No — it's a best-effort scrape tuned to this league's page template, not a 
  universal esportsdesk importer. It'll likely work on leagues using the same
  esportsdesk template, but several assumptions will break elsewhere:

  What's reasonably portable (driven by data, not hardcoded):
  - clientID/leagueID parsed from any esportsdesk URL.
  - Team names, schedule, and stats are all anchored on the actual team names
  scraped from that league — no colors are hardcoded.
  - Pagination, the PTS=G+A integrity guard, and the playoff/regular split are
  generic.

  What's fragile and league-specific:
  - Team-name detection assumes the name is a single word right before a
  19w-10l-8t record on rosters.cfm. Multi-word names ("Red Wings"), different
  record formats, or a different roster layout → wrong/missing team names, which
  then cascade (schedule/stats anchor on those names).
  - Schedule parsing relies on the exact "dateless highlights block +
  date-grouped list with date headings in separate elements" structure and the
  RS/PO prefixes. A different schedule template won't parse.
  - Stats parsing depends on the flat row order rank Name # Pos Team GP G A PTS 
  P/G PIM. A league with different columns (e.g., no P/G, or extra columns)
  won't match the regex.
  - Date year inference reads a 2025-2026 season label; a different label format
  falls back to the current calendar year.
  - Positions are never available (always defaults to Forward).

  So: same-template leagues — good odds. Arbitrary esportsdesk sites — expect
  the schedule/stats parsers to need tweaking in esportsdesk.ts. The good news
  is the failures are safe: a parse mismatch makes a phase return zero rows
  (caught → "schedule/stats failed" message) rather than importing garbage, and
  the integrity guards (PTS=G+A, no-game-over-its-score, exact standings) would
  surface bad data rather than silently accept it.

  If you want it to handle the other LCC league (31597) or genuinely arbitrary
  sites robustly, the next step would be to test against those pages and
  generalize the three parsers — but I'd treat that as a follow-up, not part of
  this branch. Want me to point it at 31597 and see what breaks?

❯ How does an import work, does it create a league or does the league have to   
  exist first?                                                                  
⏺ It creates a brand-new league — nothing has to exist first. It's a
  self-contained, one-time migration: you paste an esportsdesk URL and it builds
  the whole thing from scratch.

  The flow (manager-only, at /import):

  1. Paste URL → Preview. You give it any esportsdesk page URL (it just needs
  clientID + leagueID). It scrapes and shows you what it found — teams, player
  count, game count — plus a Season dropdown if the league has multiple seasons.
  2. Name it → Import. You confirm a league name (pre-filled with the scraped
  name) and a season name, pick the season, and hit Import.

  What it creates, in one shot:
  - a new league (public),
  - a new season under it — created inactive (is_active: false),
  - all the teams (+ enrolls them in the season),
  - all the players (+ roster rows with jersey/captain/position),
  - all the games as finalized results (so standings compute),
  - the player stat lines (distributed onto game rosters).

  It never attaches to or modifies an existing league — every run mints fresh
  records.

  A couple of consequences worth knowing:

  - It won't disturb your live league. Because the new season comes in inactive,
  the public site keeps showing whatever's currently active. To actually see
  the imported data, you switch to the new league (header league-switcher)
  and/or set its season active when you're ready.
  - Re-running is guarded. The league slug is unique, so importing the same name
  twice fails with "a league named X already exists." That's deliberate — it
  stops you from silently creating duplicate leagues. If an import half-finishes
  (e.g. schedule fails), the error message tells you to delete that league and 
  re-run, rather than re-running on top of it.
  - A few things you set afterward: activating the season, and any goalie
  positions esportsdesk didn't record (it imports positions when present,
  defaults the rest to Forward).

  So: no pre-existing league required — paste a URL, review, import, and you've
  got a new inactive league ready to activate.

❯ Why is separate instances the preferred solution to unrelated leagues?        

⏺ Because for unrelated leagues, the thing you actually want is a hard trust 
  boundary, and separate instances make that boundary structural instead of
  something you have to re-prove in every line of SQL you'll ever write.

  Here's the concrete reasoning:

  1. Isolation stops being a correctness problem. With separate Supabase
  projects, League A and League B live in different databases with different
  credentials. There is no query that can return the other org's data — it's not
  in the database. With one shared instance, isolation depends on every RLS
  policy correctly joining to the right league_id, with no bugs, forever. The
  blast radius of a single missed league check is "a different organization can
  read or edit your data" — that's a cross-tenant breach, the worst kind.
  Separate instances make that entire class of bug impossible.

  2. The burden is permanent and grows with the schema. It's not just the ~16
  policies that exist today. Every new table, view, RPC, and storage rule you
  add from now on has to remember the league scoping. This app is in active 
  growth — the importer alone just added tables. In a multi-tenant instance,
  every future migration carries a "did I scope this to league?" footgun.
  Separate instances: you never think about it again.

  3. Storage and edge cases are genuinely awkward to scope. RLS joins are clean
  for normal tables, but the logo storage policies, for example, can't easily
  join to a league_id (the exploration confirmed this) — you'd need a
  bucket-per-league or a path convention or a service-role handler. There's a
  long tail of these. Separate projects sidestep all of it.

  4. Operations map to the real boundary. Separate projects give you per-org
  backups, quotas, point-in-time restore, pausing, and billing for free. If a
  league leaves, you hand them their project or delete it — done. In a shared
  instance, cleanly extracting or deleting one tenant's data (e.g. a "give me my
  data and remove me" request) is careful surgery across a dozen FK-linked
  tables.

  5. Auth gets simpler, not harder. Unrelated orgs → separate auth pools,
  separate magic-link senders, no cross-tenant identity. The shared-instance
  route forces the opposite: the JWT has to carry a per-league role map, the
  auth hook gets more complex, and a single auth.users pool spans organizations
  that shouldn't know about each other.

  The deciding principle: multi-tenant RLS is the right tool when one trusted 
  operator intentionally hosts many tenants together and wants shared 
  infrastructure or cross-tenant features — that operator accepts the
  RLS-correctness burden in exchange for not running N projects and being able
  to build things like a shared player directory. "Unrelated leagues with
  different managers who don't trust each other" is the opposite situation: the
  trust boundary is hard, so the isolation should be hard too.

  When the shared instance genuinely wins (to be balanced):
  - It's a product/SaaS where signups create leagues on the fly and you can't
  spin up a project per signup.
  - You want cross-league features — shared player identities, a unified public
  site, one account managing several leagues.
  - You're a single operator who wants one deploy to monitor and back up, and N
  projects' overhead is worse than the RLS tax.

  The honest cost of separate instances is more deployments to babysit (though
  it's one codebase with different env vars — supabase db push against a new
  project is a single command, and the repo makes each setup reproducible) and
  no cross-league anything — but cross-league features are, by definition, not
  what you want for unrelated orgs.

  And note where this app already sits: it's multi-league within one instance
  today (everything's league_id-scoped, there's a switcher) — which is exactly
  right for one operator running several of their own leagues. The
  per-league-roles work only becomes necessary when you cross into "different,
  untrusted managers sharing one instance" — and at that point you should be
  sure you actually want a shared instance, because separate projects would get
  you stronger isolation for less code.
